深度思考
```用户询问为什么抛出异常后事务没有回滚，从日志可以看到事务被提交了而不是回滚。
这是因为 Spring 的 @Transactional 注解默认只对运行时异常（RuntimeException）和 Error 进行回滚，
而用户代码中抛出的是受检异常（Exception），所以需要设置 rollbackFor 属性来指定回滚的异常类型。

```
从你提供的日志和代码来看，事务**没有回滚反而提交了**，原因在于：

## 核心问题：Spring 事务默认只对 RuntimeException 回滚

你的代码抛出的是 **`Exception`（受检异常）**：

```java
if(true){
    throw new Exception("出现异常了~~~");  // ← 这是 Exception，不是 RuntimeException
}
```


### Spring 事务的回滚规则

```java
@Transactional
public void save(Emp emp) {
    // ...
    throw new Exception("出现异常了~~~");  // ❌ 不会回滚（受检异常）

    throw new RuntimeException("出错了");  // ✅ 会回滚（运行时异常）
}
```


**默认行为**：
- ✅ **回滚**：`RuntimeException` 及其子类、`Error`
- ❌ **不回滚**：`Exception` 及其子类（受检异常）

## 解决方案

### 方案 1：让 @Transactional 对 Exception 也回滚（推荐）

在方法上添加 `rollbackFor` 参数：

```java
@Override
@Transactional(rollbackFor = Exception.class)  // ← 添加这个
public void save(Emp emp) {
    //1.补全基础属性
    emp.setCreateTime(LocalDateTime.now());
    emp.setUpdateTime(LocalDateTime.now());

    //2.保存员工基本信息
    empMapper.insert(emp);

    //模拟：异常发生
    if(true){
        throw new Exception("出现异常了~~~");  // 现在会回滚了
    }

    //3. 保存员工的工作经历信息 - 批量
    Integer empId = emp.getId();
    List<EmpExpr> exprList = emp.getExprList();
    if(!CollectionUtils.isEmpty(exprList)){
        exprList.forEach(empExpr -> empExpr.setEmpId(empId));
        empExprMapper.insertBatch(exprList);
    }
}
```


### 方案 2：抛出 RuntimeException（修改异常类型）

```java
@Override
@Transactional
public void save(Emp emp) {
    // ...

    //模拟：异常发生
    if(true){
        throw new RuntimeException("出现异常了~~~");  // ← 改为 RuntimeException
    }

    // ...
}
```


### 方案 3：手动回滚

```java
import org.springframework.transaction.interceptor.TransactionAspectSupport;

@Override
@Transactional
public void save(Emp emp) {
    try {
        //1.补全基础属性
        emp.setCreateTime(LocalDateTime.now());
        emp.setUpdateTime(LocalDateTime.now());

        //2.保存员工基本信息
        empMapper.insert(emp);

        //模拟：异常发生
        if(true){
            throw new Exception("出现异常了~~~");
        }

        //3. 保存员工的工作经历信息 - 批量
        Integer empId = emp.getId();
        List<EmpExpr> exprList = emp.getExprList();
        if(!CollectionUtils.isEmpty(exprList)){
            exprList.forEach(empExpr -> empExpr.setEmpId(empId));
            empExprMapper.insertBatch(exprList);
        }
    } catch (Exception e) {
        // 手动标记事务回滚
        TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
        throw e;
    }
}
```


## 为什么这样设计？

Spring 的设计哲学：
- **RuntimeException**：程序错误，通常是 bug，应该回滚
- **Exception（受检异常）**：业务异常，可能是可预期的，不一定需要回滚

例如：
```java
// 账户余额不足 - 业务异常，可能不需要回滚整个事务
throw new InsufficientBalanceException("余额不足");

// 数据库连接失败 - 系统错误，必须回滚
throw new RuntimeException("数据库异常");
```


## 最佳实践

在实际开发中，**推荐对所有异常都回滚**：

```java
@Transactional(rollbackFor = Exception.class)
public void save(Emp emp) {
    // 这样无论是 RuntimeException 还是 Exception 都会回滚
}
```


或者在类级别统一配置：

```java
@Service
@Transactional(rollbackFor = Exception.class)  // ← 类级别配置
public class EmpServiceImpl implements EmpService {

    @Override
    public void save(Emp emp) {
        // 所有方法都会对 Exception 回滚
    }
}
```
